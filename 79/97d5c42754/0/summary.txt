1️⃣ `Tier 1` / `Tier 2` ピンの地図上での混在表示デザイン

---

🔍 1️⃣ ピンのビジュアルランゲージ

まずこれ、前回の設計書（P3）でパーク vs ストリートの区別を【色 + 形状 + アイコン】でやるっていうアラインメントは取れてるんだよね。今回そこに `Tier 1` / `Tier 2` のレイヤーが加わるわけだけど、ここで情報のヒエラルキーを整理しないと、ユーザーのコグニティブロードが爆発する。

僕の提案はこう:

📋 第1軸（形状 + アイコン）: スポットタイプ
- パーク → 丸みのあるピン + スケートパークアイコン
- ストリート → 角張ったピン + ストリートアイコン

📋 第2軸（透明度 + サイズ）: データティア
- `Tier 1`（マスター）→ フルカラー、標準サイズ。メインコンテンツとしてのプレゼンス
- `Tier 2`（ユーザー個人）→ やや小さめ、もしくはセカンダリカラー。「自分だけのメモ」感

ポイントは、`Tier 2` を目立たせすぎないこと。SpotDiggzのコアバリューは `Tier 1` マスターデータによる「信頼できるスポット発見」なんだよね。`Tier 2` はあくまでパーソナルなサブ機能。ここのプライオリティをビジュアルでも明確にしないと、ユーザーが「どれが公式の情報で、どれが自分のメモか」わからなくなる。

具体的なイメージ:

```
〔地図上のピン表示〕

  🛹  ← Tier 1 パーク: フルカラー、標準サイズ、リッチなアイコン
  🔶  ← Tier 1 ストリート: フルカラー、標準サイズ

  📌  ← Tier 2（ユーザー）: シンプルなピン、小さめ、グレー系 or パステル
       タップするとラベルに「マイスポット」と表示
```

あと、フィルタリングのオプションとして「マイスポットだけ表示」「マスターデータだけ表示」のトグルがあるとユーザーエクスペリエンスのカスタマイズ性が上がる。これはP1-1の検索・フィルタ設計に組み込める。

---

🔍 2️⃣ マイリストのローカル移行

マイリストがFirestoreからSwiftData + CloudKitに移行するわけだけど、ユーザーサイドから見たエクスペリエンスへのインパクトを整理するね。

📋 ポジティブなインパクト:
- オフラインでもマイリストが見られる。これはカスタマーバリューとして大きい
- レスポンスがローカル読み取りになるから、表示が速い。スナッピーなUX
- iCloud同期で端末間の引き継ぎも自然にできる

📋 ネガティブなインパクト / ケアすべき点:
- 既存ユーザーのマイリストデータがFirestoreにある場合、マイグレーションが必要
  → ……ただ、現時点でアクティブユーザーがどの程度いるかによるね。まだリリース前なら問題ない
- iCloudオフのユーザーは、端末変更時にマイリストが失われる
  → これは `Tier 2` スポットと同じ注意表示でカバーできる

📋 UIの変更点:
- マイリストの操作（追加・削除）がAPI経由からローカル操作に変わるので、ネットワーク待ちがなくなる
- 「お気に入りに追加」のフィードバックが即座に反映される（ハプティックフィードバック + アニメーション推奨）
- プロフィール画面のマイリスト表示は変更なし。データソースが変わるだけ

---

📋 サマリー

| 項目 | 玉縄の見解 |
|:--|:--|
| Tier 1/2 混在表示 | OK。ただしビジュアルヒエラルキーで Tier 1 を優先表示すること |
| ピン区別方法 | 第1軸: スポットタイプ（形状）、第2軸: ティア（サイズ・カラー） |
| Tier 2 の表示 | 控えめ。「マイスポット」ラベルで自分のメモであることを明示 |
| フィルタ追加 | 「マイスポット表示/非表示」トグルを推奨 |
| マイリスト移行 | UXはポジティブ。オフライン対応 + 高速化。UI変更は軽微 |

うん、全体のディレクションとしてバリューが出てると思うんだよね。`Tier 1` のプレゼンスをしっかり維持しつつ、`Tier 2` はパーソナルなレイヤーとしてブラッシュアップしていこう。

---

……直斗に戻ります。

玉縄の見解をまとめると、`Tier 1` / `Tier 2` の混在表示は【ビジュアルヒエラルキーを守れば問題ない】。マイリストのローカル移行もUX面ではポジティブという判断です。

これで未決事項は全て解消しました。設計書を更新して、次のステップ——`/design` による詳細設計、もしくは直接 Plan Mode での実装計画に進めます。どちらがよいですか？
